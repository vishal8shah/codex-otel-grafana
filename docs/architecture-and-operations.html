Architecture and operations

Codex emits OTLP signals to a local Collector. The Collector applies redaction and routing before Loki, Tempo, metrics storage, and Grafana. Diagnostics are added only when the relevant fields are confirmed.

Signal roles

Logs

Confirmed event records provide completion, activity, model, tool, and timing evidence where recorded.

Traces

Observed spans provide execution context. A trace attribute is not joined to a run unless that correlation is proven safe.

Metrics

Collector health and derived spanmetrics verify pipeline behavior. They are not native codex_* metrics.

Codex Stuck Triage derivation

Raw records are grouped using a hash of the source conversation identifier. The raw identifier is discarded; derived codex.run_health rows expose only run_hash, state, safe timing, model, last event, and explanatory fields.

schema-confirmed Codex logs
  -> hash source identifier
  -> classify completion and quiet-time evidence
  -> emit derived codex.run_health
  -> query unique run_hash values in Grafana

A stuck state is a heuristic candidate. It does not prove a Codex product bug and does not claim to fix one.

Privacy boundary

Prompt logging is disabled. The collector drops known identity and prompt-like attributes, and the analyzer uses an allowlist before emitting derived records.

Never expose

Raw prompts, identities, conversation identifiers, local paths, raw endpoints, raw call IDs, tool arguments, and tool output.

Safe diagnostic fields

Privacy-safe run hash, derived state, quiet time, last event, model where schema-backed, tool name where schema-backed, endpoint hash, status bucket, bounded duration and attempt evidence, retry/slow flags, and bounded source window.
