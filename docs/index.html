Why use it

Move from “something feels off” to a bounded investigation.

The kit is useful precisely because it refuses to make the leap from evidence to certainty.

01

Stop guessing

See whether recent evidence points toward a quiet run, tool result, API request group, or slow confirmed contributor.

02

Keep private work private

Group with hashes and allowlisted fields rather than exposing prompts, identities, paths, arguments, or output.

03

Open the focused view

Use the Command Center first, then follow the linked dashboard into the relevant privacy-safe grouping and source window.

04

Keep claims bounded

Every diagnostic says what its telemetry can support—and what remains unknown, aggregated, or unproven.

Symptom
Codex feels unreliable or fails around model/backend activity.
Inspect
State, status bucket, maximum duration, maximum attempt, retry flag, endpoint hash, run hash, and source window.
This evidence suggests
A run/endpoint group contains failed, retried, slow, successful, or incomplete request evidence.
Next action

Symptom

Inspect
Contributor type, duration, configured threshold, run hash, endpoint hash or tool name, and source window.

Privacy boundary

Useful evidence crosses. Private work does not.

Raw identifiers are hashed or dropped before derived records are used by diagnostic dashboards.

×

DO NOT EXPOSE

Unsafe raw fields

  • Prompts and identities
  • Raw conversation identifiers
  • Raw endpoints and local paths
  • Raw call identifiers
  • Tool arguments and tool output
HASHDROPbefore grouping
✓

ALLOWLISTED

Privacy-safe derived evidence

  • run_hash and endpoint_hash
  • tool_name where schema-backed
  • Derived state and status buckets
  • Bounded timing, retry, and source-window evidence
  • Last event type where safe

Proof timeline

Ten shipped milestones. Every claim earned.

Each cycle records both the proof and the line it deliberately did not cross.

Phase0

Schema evidence Shipped

Proven: observed fields, privacy boundaries, and explicit unknowns.

Not claimed: documentation-only fields as runtime proof.

Phase0b

Fresh interactive evidence Shipped

Proven: version-stamped interactive Codex telemetry evidence.

Not claimed: invalid entrypoints or unobserved native metrics.

Phase1

Local LGTM Shipped

Proven: reproducible local Collector, Loki, Tempo, metrics storage, and Grafana.

Not claimed: hosted or production operations.

Phase1b

File-provisioned dashboards Shipped

Proven: repository-owned datasource and dashboard provisioning.

Not claimed: manually curated UI state as source of truth.

Phase2

Stuck Triage Shipped

Proven: unique-run investigation evidence from codex.run_health.

Not claimed: token burn without schema-backed completion evidence.

Phase3

Tool Failure Diagnosis Shipped

Proven: run/tool-level result evidence from confirmed fields.

Not claimed: individual-call proof.

Phase4

API Request Reliability Shipped

Proven: failed, retried, slow, and incomplete run/endpoint evidence.

Not claimed: raw endpoints or individual-request proof.

Phase5

Slow Contributor Triage Shipped

Proven: slow confirmed API and tool contributors.

Not claimed: full end-to-end turn latency.

Phase6

Stuck Candidate Notification Shipped

Proven: opt-in local/dev push path from derived run-health evidence.

Not claimed: production alert management or always-on trustworthiness.

Phase7

Diagnostic Command Center Current

Proven: one front door over the four shipped derived streams.

Not claimed: ranking, raw classification, onboarding, or production monitoring.

Claim boundaries

Honesty is part of the interface.

Full capability ledger →
NO

No native codex_* Prometheus metrics were observed in the valid interactive evidence run.

NO

No raw prompts, identities, conversation IDs, paths, endpoints, tool arguments, or tool output.

NO

No diagnostic result proves a Codex bug—and notification silence does not prove health.

NO

No production alerting or monitoring claim. Phase 6 is a local development proof path.

NO

No automatic ranking. Phase 7 surfaces categories and routes to focused views.

NO

No runtime proof in CI. CI remains static and unit; per-diagnostic runtime proof is manual.
