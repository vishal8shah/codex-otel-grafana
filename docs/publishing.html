  • The displayed public brand is Codex Observability Diagnostic Kit; observability describes the OpenTelemetry and Grafana foundation, not a broad platform or complete coverage.
  • No secrets, raw prompts, identities, conversation identifiers, call identifiers, raw endpoints, paths, tool arguments, or tool output.
  • No decorative dashboard totals, fixed cost results, or unverified issue references.
  • Every committed dashboard screenshot uses synthetic/local proof data, carries a visible synthetic-example label and matching alt text, and has a completed entry in dashboard-walkthrough-privacy-review.md.
  • Every diagnostic is tagged Shipped, Next, Backlog, Acknowledged, Partial, or Not claimed.
  • Stuck, tool, API, and slow-contributor triage are shipped issue-led diagnostics.
  • Token burn without completion and full end-to-end turn latency are explicitly not claimed.

Suggested public summary

When Codex goes quiet, developers should not have to guess.

The Codex Observability Diagnostic Kit provides local-first OpenTelemetry and Grafana observability for Codex diagnostics. It turns schema-confirmed telemetry into privacy-safe derived evidence.

Phases 2-7 ship Stuck Triage, Tool Failure Diagnosis, API Request Reliability, Slow Contributor Triage, opt-in stuck-candidate notification, and a derived-evidence Diagnostic Command Center. The Command Center is a light front door for users of the running local kit; it is not onboarding, alerting, or production monitoring. The slow view covers confirmed API/tool contributors, not full-turn latency. Notification depends on fresh derived run-health records and is not production alert management. Review/resume flow diagnosis is the next backlog cycle.
