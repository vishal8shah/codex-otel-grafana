Cross-platform local setup

Quickstart and manual rebuild

Start the repository-owned LGTM stack, verify local connectivity, run the privacy-safe analyzer, and stop without deleting retained data.

Start and verify

Windows PowerShell

.\scripts\start.ps1 -Pull
.\scripts\doctor.ps1
.\scripts\schema-verify.ps1

macOS / Linux

./scripts/start.sh --pull
./scripts/doctor.sh
./scripts/schema-verify.sh

Grafana is available at http://localhost:3000. Dashboards and datasources are provisioned from repository files when the stack starts.

Codex OpenTelemetry config

Telemetry routing belongs in the user-level Codex config. Keep prompt logging disabled and use inline exporter tables.

[otel]
environment = "local"
log_user_prompt = false
exporter = { otlp-http = { endpoint = "http://localhost:4318/v1/logs", protocol = "binary" } }
trace_exporter = { otlp-http = { endpoint = "http://localhost:4318/v1/traces", protocol = "binary" } }
metrics_exporter = { otlp-http = { endpoint = "http://localhost:4318/v1/metrics", protocol = "binary" } }

Use interactive codex, not codex exec, for telemetry schema discovery. Submit only a tiny non-sensitive prompt.

Choose where to start

Open Grafana > Codex Diagnostic Command Center after the local kit is running. Its shared lookback defaults to six hours to match the focused dashboards. It summarizes existing privacy-safe derived streams and links to Stuck Triage, Tool Failure Diagnosis, API Request Reliability, and Slow Contributor Triage.

This is a light navigation front door, not an onboarding or installation flow. It adds no classifier, reads no raw Codex telemetry directly, does not replace Phase 6 notification, and is not production monitoring. Zero or missing derived records do not prove health.

Run Codex Stuck Triage

Windows PowerShell

.\scripts\run-health.ps1
.\scripts\run-health.ps1 -EmitDerived

macOS / Linux

./scripts/run-health.sh
./scripts/run-health.sh --emit-derived

Open Grafana > Codex Stuck Triage. The stat panels count unique run_hash values over the selected range; the table is snapshot-based and may show repeated analyzer emissions.

Opt in to stuck notification

Windows PowerShell

python tools/alerting/dev_webhook_listener.py --host 0.0.0.0
.\scripts\watch-stuck.ps1 -EmitDerived

macOS / Linux

python3 tools/alerting/dev_webhook_listener.py --host 0.0.0.0
./scripts/watch-stuck.sh --emit-derived

The watcher is opt-in, defaults to dry-run, and is not started by the stack. Grafana alerts only on recent derived codex.run_health records; it does not classify raw Codex telemetry. The listener defaults to 127.0.0.1. Docker proof explicitly uses --host 0.0.0.0 because containerized Grafana reaches the host through host.docker.internal; loopback-only binding cannot receive that webhook. This is temporary development-proof widening, not production exposure.

On Windows, a Defender Firewall prompt may appear for 0.0.0.0:9087. Allow only the private profile, avoid public networks, scope the rule to the Docker subnet when practical, time-box the proof, and stop the listener afterwards.

The complete chain must be healthy: LGTM/Grafana, incoming Codex telemetry, emitting watcher/analyzer, provisioned rule, contact-point routing, and reachable destination. noDataState=OK and execErrState=OK are fail-open for development proof. Silence may mean healthy, but it may also mean the analyzer stopped, Loki/query evaluation failed, provisioning or routing failed, or the destination was unreachable. Silence does not prove Codex is healthy. The alert is investigation evidence, not proof of a Codex bug.
