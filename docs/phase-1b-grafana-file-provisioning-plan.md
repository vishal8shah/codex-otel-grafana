# Phase 1b: Grafana File Provisioning Plan

## Purpose

Phase 1b should make the provisioned Grafana state reproducible across Windows,
macOS, Linux, and WSL2 without requiring PowerShell. This document defines the
migration and validation approach; it does not implement provisioning or change
dashboard behavior.

## Current State

- Docker Compose starts the local `grafana/otel-lgtm` stack cross-platform and
  mounts only the OpenTelemetry Collector configuration from this repository.
- Dashboard provisioning remains PowerShell-based. The
  `observability/setup-codex-dashboards.ps1` script constructs four dashboards
  in memory and publishes them through the Grafana HTTP API.
- No dashboard JSON files currently exist under `observability/`.
- The script relies on the existing Grafana datasource UIDs `loki`, `tempo`, and
  `prometheus`; it does not create those datasources.
- macOS and Linux users need `pwsh` to reach the same dashboard state as the
  Windows workflow.

Phase 1b should remove this operating-system asymmetry while preserving the
existing dashboards, UIDs, queries, folder, and local-first boundaries.

## Proposed Target

- Define the required Grafana datasources in repository-owned provisioning
  files with stable names and UIDs.
- Export the four existing dashboards to repository-owned JSON without changing
  their logic.
- Define a dashboard provider that loads those JSON files into the existing
  `Codex Observability` folder.
- Mount provisioning directories and dashboard JSON directories read-only from
  Docker Compose.
- Make `docker compose up` converge Windows, macOS, Linux, and WSL2 on the same
  Grafana datasource and dashboard state.

The implementation should first confirm the provisioning and dashboard paths
supported by the tested `grafana/otel-lgtm` image. Repository mounts must not
replace unrelated image-owned provisioning unintentionally.

## Candidate File Structure

```text
observability/
  provisioning/
    datasources/
      datasources.yml
    dashboards/
      dashboards.yml
  dashboards/
    codex-logs.json
    codex-traces.json
    codex-prometheus-spanmetrics.json
    codex-token-economics.json
```

Candidate container mounts:

```text
./observability/provisioning/datasources -> Grafana datasource provisioning path (read-only)
./observability/provisioning/dashboards  -> Grafana dashboard-provider path (read-only)
./observability/dashboards               -> provider dashboard path (read-only)
```

Exact container destinations are an implementation-time proof item and must be
verified against the pinned LGTM image rather than assumed.

## Migration Approach

1. Pin `grafana/otel-lgtm` to a tested tag and record the Grafana version and
   provisioning paths used by that image.
2. Add explicit datasource definitions with stable UIDs matching the dashboard
   references: `loki`, `tempo`, and `prometheus`.
3. Export the four dashboards generated by
   `setup-codex-dashboards.ps1` into deterministic JSON files. Compare titles,
   UIDs, folder placement, variables, panels, queries, refresh interval, and
   time range; do not redesign or normalize dashboard logic during extraction.
4. Add the dashboard provider and read-only Compose mounts alongside the
   existing collector mount.
5. Keep the PowerShell script temporarily as a known comparison and manual
   refresh path.
6. Validate file-provisioned parity on retained and clean test volumes.
7. Decide in a later change whether to retire the PowerShell script or retain it
   as an optional manual refresh/export tool.

The file-based definitions should become the source of truth before the
PowerShell path is removed. Avoid maintaining two independently editable copies
of dashboard logic.

## Risks and Proof Questions

- **LGTM paths:** The image may preconfigure Grafana paths or mount layouts that
  differ from a standalone Grafana container.
- **Image defaults:** Replacing a whole provisioning directory could hide or
  remove LGTM-provided datasources. Prefer additive file mounts or a verified
  directory overlay strategy.
- **Dashboard UID collisions:** The retained volume may already contain the same
  dashboard UIDs. Provisioning behavior and ownership must be confirmed.
- **Datasource identity:** Datasource names, UIDs, types, URLs, access mode, and
  trace/log correlations must match dashboard references and LGTM networking.
- **JSON compatibility:** API payloads produced by the PowerShell script may
  need their outer publish envelope removed before use as provisioned dashboard
  JSON. Grafana schema-version compatibility must be checked.
- **Folder behavior:** The provider must reliably place dashboards in `Codex
  Observability` without creating duplicate folders or dashboards.
- **Retained-state masking:** Existing datasources and dashboards in the named
  volume can make broken provisioning appear successful.
- **Clean-volume safety:** Clean-volume testing must use a disposable test volume
  or an explicit backup/restore plan. The existing local data volume must not be
  deleted as an incidental validation step.
- **Update semantics:** Confirm whether provisioned dashboards are editable in
  the UI and how file updates overwrite or reconcile retained state.

## Validation Plan

### Static checks

1. Run `docker compose config` and inspect resolved read-only mounts.
2. Validate provisioning YAML and dashboard JSON syntax.
3. Compare extracted dashboard UIDs and datasource UIDs with the existing
   PowerShell definitions.

### Retained-volume runtime

1. Run `docker compose up -d` with the retained named volume.
2. Confirm Grafana health through `/api/health`.
3. Query the Grafana API to confirm the expected datasources, their stable UIDs,
   and their health.
4. Confirm all four dashboards, expected UIDs, and `Codex Observability` folder.
5. Compare representative panels and queries with the PowerShell-produced
   dashboards.
6. Stop and remove the Compose container and network while preserving the named
   volume.

### Clean test-volume runtime

1. Use a separate, clearly named disposable test volume or isolated Compose
   project; do not purge `codex-otel-lgtm-data`.
2. Start from empty Grafana state and repeat health, datasource, folder, and
   dashboard checks.
3. Restart the stack to verify provisioning is idempotent and does not create
   duplicates.

### Platform checks

- Run the retained- or test-volume workflow through Windows PowerShell.
- Run the same workflow through WSL2 Bash.
- Record macOS/Linux verification separately if those hosts are available; do
  not infer host-specific success from syntax checks alone.

## Explicit Non-goals

- No new dashboard logic or panel changes.
- No synthetic telemetry.
- No changes to `SCHEMA.md` or observed schema findings.
- No assumption that native `codex_*` Prometheus metrics exist; they remain
  optional until observed in a future schema run.
- No Phase 2 issue-scenario documentation.
- No production-readiness claim, public service exposure, or secret management
  expansion.
- No alerting or rule provisioning until dashboard and datasource parity is
  proven. Alerting may be designed as a separate follow-up if justified.

## Follow-up Cleanup Items

- Pin `grafana/otel-lgtm` to a tested image tag and document the update and
  rollback process.
- Replace `start.sh` parsing of human-readable `docker compose config` output
  with robust JSON parsing. Prefer `docker compose config --format json`, use
  `jq` when available, and provide a documented safe fallback rather than making
  `jq` an unexplained hard dependency.
- Split `doctor.ps1` and `doctor.sh` into stack-health checks that determine the
  default exit code and informational Codex CLI/config readiness checks. A
  future explicit strict/full mode may require both groups to pass.

## Decision Gate for Implementation

Implementation should begin only after confirming the tested LGTM image tag,
its Grafana provisioning paths, the internal datasource URLs, and the strategy
for generating deterministic dashboard JSON from the current PowerShell source.
The first implementation review should demonstrate parity, not visual or query
improvements.
